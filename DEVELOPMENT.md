# Development

## Tech stack

- [SvelteKit](https://kit.svelte.dev/) + TypeScript
- [Drizzle ORM](https://orm.drizzle.team/) + Postgres (`postgres-js` driver against [Neon](https://neon.com))
- [PGlite](https://pglite.dev/) for in-process Postgres in tests (no Docker)
- [marked](https://marked.js.org/) for Wiki markdown preview

## Setup

**Prerequisites:** Node.js 24+, npm. A Postgres connection — either a Neon
dev branch or local Postgres via Docker.

```sh
npm install
cp .env.example .env       # set DATABASE_URL (see below)
npm run db:migrate
npm run dev
```

`DATABASE_URL` shapes:

```
# Neon dev branch (recommended)
DATABASE_URL=postgres://user:pass@ep-xxxxx.region.aws.neon.tech/dbname

# Local docker (offline fallback — see docker-compose.yml)
DATABASE_URL=postgres://dev:dev@localhost:5432/betwixt
```

## Database

```sh
npm run db:push      # apply schema changes directly (dev shortcut)
npm run db:generate  # generate a new migration from schema changes
npm run db:migrate   # apply all pending migrations
npm run db:studio    # open Drizzle Studio to browse the database
```

The schema is generated by Drizzle from `src/lib/server/db/schema.ts`. The
baseline migration `drizzle/0000_strong_martin_li.sql` includes a
hand-added `bump_updated_at()` BEFORE UPDATE trigger that keeps
`updated_at` fresh on every UPDATE — application code does not set this
column.

### Local Postgres via docker-compose

```sh
docker compose up -d
DATABASE_URL=postgres://dev:dev@localhost:5432/betwixt npm run db:migrate
```

## Testing

All tests live under `tests/`, split into three buckets:

```
tests/
  unit/          Pure logic, no DB. Mocked fetch.
  integration/   DB-backed (PGlite in-process) + API handler tests via vi.mock.
  e2e/           Playwright against the running preview server.
  helpers/       Shared test fixtures (test-db.ts).
```

Vitest is configured with two named projects (`unit` and `integration`)
that share a base config but match different glob patterns.

```sh
npm test                              # all Vitest (unit + integration)
npm test -- --project unit            # fast, no DB
npm test -- --project integration     # DB-backed via PGlite, ~5s
npm run test:e2e                      # Playwright, ~60s incl. build
```

Integration tests use [PGlite](https://pglite.dev/) — Postgres compiled
to WASM, in-process, same SQL/types/planner/transaction semantics as
full Postgres. No Docker, no Testcontainers, no external dependencies.
The shared factory at `tests/helpers/test-db.ts` reads + applies the
actual `drizzle/*.sql` migration files on each fresh test DB, so a drift
between schema and migration is caught at test time, not at deploy time.

E2E tests build and serve a production preview (`npm run build && npm
run preview`) on port 4173, then run serially (`workers: 1`) because all
tests share a single Postgres database.

```sh
# Install Playwright browsers (once)
npx playwright install firefox

npm run test:e2e
```

**Coverage**: ~382 Vitest tests across 33 files. See `tests/unit/` for
pure logic + store behavior; `tests/integration/` for migrations,
invariants, cascade chains, API handlers, jsonb round-trip, BEFORE
UPDATE trigger, splitInterval atomicity under tx rollback, uuid PK
generation, CHECK constraint enforcement.

## Deploying

The current build targets [Fly.io](https://fly.io) with `adapter-node`
against a Neon Postgres database. CONSIDERATIONS.md → "[2026-05-01]"
captures the locked deployment stack. Auth + deploy infra lands in T8b
after Phase 1B graph features ship.

Until T8b ships, `npm run dev` against a Neon dev branch is the
deploy-ready path. The Postgres baseline is identical between dev and
prod; only the URL changes.
