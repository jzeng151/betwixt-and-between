## Problem Statement

Plan the next wave of features for betwixt-and-between (Wiki rework / T6, responsive World Map, standalone Notes app, T8b deploy + auth) so they can be developed on independent branches with minimum merge conflicts and a stated merge protocol for the unavoidable shared files. Personal worldbuilding tool first; multi-author real-time collab is a future arc that the architecture must not preclude.

Honest framing on parallelism: there is no such thing as "zero conflict" parallel branches in a small SvelteKit app — `package.json`, `lib/types.ts`, `+layout.svelte`, and Drizzle's migration journal are all shared. Goal here is to make each branch's writes to those files **trivially mergeable** through a stated protocol, not to pretend the conflicts don't exist.

## What Makes This Cool

The 10x insight: **the playhead is the central nervous system.** Every feature on the slate is reaching for the same primitive — a current-scope reactive store derived from the playhead — and once you build it once, every window in the app becomes responsive to where you are in the story.

- Wiki sidebar dims out-of-scope entries (the deferred "5.12" subscriber)
- World Map regions glow / dim by scope
- Palette filters to in-scope characters/events
- StoryGraph fades out-of-scope nodes
- Future collab presence rides the same store ("Alice is at Act 3 / Scene 2")

Plus a paired primitive — a **preferences store** — that lives in localStorage now and migrates to a `users.preferences jsonb` column once T8b lands. Theme + entity defaults + (eventually) hotkeys all subscribe.

The world map's design absorbs A+C combined: imported bitmap base + polygon regions linked to Locations as Phase 1, with hex-grid-overlay-with-fog-of-war pulled out into Phase 1.5 because the row-per-hex storage model needs to be solved (see Schema section) and the playhead-driven reveal is genuinely speculative for personal-writer mode.

Three additions beyond the original slate, **explicitly Phase 1.5** (not smuggled into Phase 1):

- **Cmd-K everywhere** (Phase 1.5) — fuzzy search over entities + commands. Pairs with the hotkey registry, which is also Phase 1.5 (moved out of Phase 0 because YAGNI until Cmd-K consumes it).
- **Story Player mode** (Phase 1.5) — read-only auto-advancing playhead; the demo moment.
- **Static HTML wiki export** (Phase 1.5) — `npm run export` produces a self-contained read-only site. Coupled to the World Map bitmap-storage decision; do not start until that lands.

## Constraints

- Single developer (Jason). "Parallel branches" = independence, not concurrency. Branches must be mergeable in any order without human-fixable conflicts beyond the migration-claim protocol.
- Builder / learning / fun mode. No deadline pressure. Optimize for craft and momentum.
- Eventual multi-author collab arc. T8b architecture must not preclude future shared-document + presence + Yjs CRDT layer.
- Solo personal use today; deploy comes after enough features to make deploying worthwhile.
- Existing tech stack must be preserved: SvelteKit, Drizzle, Postgres (T8a shipped), tldraw for StoryGraph.

## Premises (revised after review)

All locked with the user during this session except where revised:

1. **The `currentScope` store is the load-bearing primitive.** Concrete shape:
   ```ts
   type CurrentScope = {
     t: number;                   // playhead position, fractional acts
     actId: string | null;
     sceneId: string | null;
     activeCharacterIds: Set<string>;
     activeEventIds: Set<string>;
     activeLocationIds: Set<string>;
     activeNoteIds: Set<string>;  // notes attached to active entities
   };
   ```
   Derived from `$playhead`. Pure function; no side effects. Subscribers re-render on shallow Set diff.

2. **A preferences foundation is co-equal Phase 0.** Hotkeys are NOT in Phase 0 — moved to Phase 1.5 (with Cmd-K). Phase 0a contains only: preferences store, Settings.svelte shell, Appearance section (light/dark/auto + accent + density), entity-default colors. Pre-T8b: localStorage. Post-T8b: server-side `users.preferences jsonb`; localStorage becomes a cache.

3. **T8b ships LAST in this arc.** Reason: every Phase 1 feature adds tables; T8b's user_id migration sweeps over everything that exists at that moment. Honest re-estimate: T8b is a **week+, not 2 days**. The 2-day estimate from CONSIDERATIONS.md was for the locked spec; with realistic test coverage, magic-link mailer wiring, OAuth provider mock, isolation-test scaffolding, and Fly.io deploy debugging, expect 5–7 working days.

4. **Notes app is the most independent feature.** Separate tables (`note_folders`, `notepad_entries`), separate component, no scope/EntityDetail dependency. Optional one-way import:
   - **Trigger:** explicit user action via Settings → Data → "Import character notes". Confirmation dialog.
   - **UI surface:** Settings panel button, also a button on each character's EntityDetail.
   - **Idempotent:** existing imports tagged with `source: { kind: 'character', entityId, importedAt }`; re-import diffs by entityId, only adds entries for characters not yet imported. User can "force re-import" which prompts for overwrite/skip per existing entry.

5. **Wiki rework's Location-branch is a soft prerequisite for World Map.** World Map regions link to Location entities. Mitigation: land `feat/wiki-location-branch` (just the EntityDetail Location editor, no other Wiki rework) as a small slice first; rest of T6 continues in parallel with Map.

6. **Cmd-K depends on the hotkey-registry, which is now Phase 1.5.** Cmd-K does NOT depend on the scope-store; it can rank by recency + simple text match without scope awareness. (Premise 6 in the v1 doc was wrong about the scope-store dep.)

7. **Hex grid + fog-of-war is Phase 1.5, not Phase 1.** Storage model: client-derived hexes from `(map_id, hex_size)` tuple — no row per hex. Persist only sparse deviations: a `map_hex_overrides` table with `(map_id, x, y, revealed_at, region_override_id)` rows for hexes that diverge from defaults. Reveals from playhead update `revealed_at` on hexes intersecting the active scene's location regions.

8. **Migration-claim merge protocol.** Drizzle's migration filenames are sequentially numbered (`0001_*.sql`, `0002_*.sql`) and the journal at `drizzle/meta/_journal.json` records the order. Any branch that adds a migration must claim its slot at merge time, not at branch time. **Protocol:** when ready to merge, the branch rebases against main, regenerates migration with `npm run drizzle:generate`, lets Drizzle pick the next available number, and force-pushes. Rebases happen sequentially across feature branches via the queue order in the Branch Plan section (Notes → Wiki → Map). The doc author or a release manager owns the queue.

## Approaches Considered

### Approach A — Foundation-first, then 5-way feature parallel

```
Phase 0  ▶  0a preferences │ 0b scope                   (2 parallel tracks)
Phase 1  ▶  Notes │ Wiki │ Map (regions)                (3 parallel)
Phase 1.5 ▶ hotkey-registry → Cmd-K │ Story Player │ static-export │ hex/fog (4 parallel after hotkey)
Phase 2  ▶  T8b
```

Textbook order. Phase 0 lands first (only 2 tracks now after dropping hotkey-registry); then features parallelize.

- **Pros:** Maximum parallelism in Phase 1. Easy to pause/resume any branch.
- **Cons:** Phase 0a's prefs-scaffold gates Notes branch start (~½ day serialization tax, not zero). If Phase 0a's preferences design stalls, Notes waits.
- **Effort:** L · **Risk:** Low

### Approach B — Vertical slice: Wiki rework first

```
Phase 0  ▶  0a │ 0b                                     (2 parallel)
Phase 1a ▶  Wiki rework solo (validates patterns)
Phase 1b ▶  Notes │ Map (regions)                       (2 parallel)
Phase 1.5 ▶ Cmd-K (after hotkey-registry) │ Story Player │ static-export │ hex/fog
Phase 2  ▶  T8b
```

Wiki rework is the most cross-cutting feature. Shipping it solo first validates patterns before others build on them.

- **Pros:** Better quality; catches design issues early.
- **Cons:** Less parallelism; ~1 week of serial work in the middle. World Map waits on the whole Wiki rework.
- **Effort:** L · **Risk:** Med

### Approach C — Pipelined: Phase 0 + Notes ride together (RECOMMENDED, revised)

```
Phase 0  ▶  0a-scaffold (½ day) ─→ 0a-app │ 0b-scope │ Notes  (3 parallel after gate)
Phase 1  ▶  Wiki │ Map (regions)                              (2 parallel)
Phase 1.5 ▶ hotkey-registry ─→ Cmd-K │ Story Player │ static-export │ hex/fog
Phase 2  ▶  T8b
```

Notes app starts as soon as the prefs-scaffold (½ day) lands. Foundations get exercised by a real consumer. Phase 1 narrows to Wiki + Map regions only.

- **Pros:** Highest parallelism after the ½-day scaffold gate. Foundations validated end-to-end by Notes. Phase 1 is focused (3 features instead of 5+).
- **Cons:** Small serialization tax up front (the prefs-scaffold gate). Notes branch coordinates lightly with 0a-app on preferences-store interface.
- **Effort:** L · **Risk:** Low

## Recommended Approach

**Approach C — Pipelined.** Reasons:

1. Honors the user's bias toward parallel-able branches.
2. Foundations get a real consumer immediately (Notes dogfoods preferences) → catches design issues during Phase 0.
3. Small serialization tax (½ day scaffold) is cheap and explicit, not hidden.
4. Phase 1 is honest: 3 features, not 5. Phase 1.5 collects the 10x stretch features with their own gate.
5. T8b's single sweeping migration covers all new tables in one pass.

**Real conflicts (not mergeable by accident):**

- `EntityDetail.svelte` — wiki-loc + wiki-rework. Resolved by sequencing: wiki-loc lands first as a small slice; wiki-rework rebases.
- `+layout.svelte` — prefs-app + notes + cmd-k all add launcher/mount entries. Resolved by **append-only protocol**: each branch appends to the apps/launchers list, never reorders existing entries. Merge becomes mechanical.
- `package.json` / `package-lock.json` — multiple branches add deps. Resolved by **lockfile-regen-on-merge protocol**: at merge time, accept incoming `dependencies` block, then `rm package-lock.json && npm install` to regenerate. Never hand-merge the lockfile.
- `lib/types.ts` — multiple branches add types. Resolved by **append-only protocol**: each branch appends new type exports to the bottom; never edits existing exports.
- `drizzle/meta/_journal.json` + migration filenames — Notes / Map / hex-fog / T8b all add migrations. Resolved by **migration-claim protocol**: at merge time, branch rebases against main, runs `npm run drizzle:generate` (Drizzle picks next available number), force-pushes, merges. Sequential merge order: Notes → Wiki → Map → (Phase 1.5) → T8b. The doc author owns the queue.

**No "✏️/✏️" collisions on feature-app files.** Each app file is owned by one branch.

## Open Questions

1. **Bitmap storage for World Map.** Local dev: filesystem under `static/maps/`. Production: undecided. Options: (a) Backblaze B2 (already in T8b backup spec — could store images same place). (b) Postgres bytea (simple but DB bloat). (c) Neon Storage if/when GA. **Lean: B2.** Decide as part of `feat/world-map` branch kickoff.
2. **Hex size: per-map or per-user-preference?** Lean per-map (stored in `world_maps.hex_size` column). Settings provides only the default for new maps.
3. **Cmd-K search index strategy.** Lean: in-memory derived from existing `/api/entities` + `/api/notes` endpoints, ranked by recency + fuzzy match (Fuse.js, ~10KB). No Postgres FTS yet. Revisit if dataset grows past ~10k entities.
4. **Static export — does it bundle the World Map bitmap?** Yes, but only in B2-backed deployments where bitmaps are URL-addressable. For local-only export, prompt user to download bitmaps separately or skip the map. Document as a known limitation.
5. **Story Player auto-advance speed.** Configurable in Settings (Editor section): seconds-per-scene, with manual play/pause/scrub controls.
6. **Folder hierarchy depth in Notes app.** Unlimited via `parent_id`; UI flattens to 2 levels at a time.
7. **Collab arc readiness check at T8b time.** T8b should NOT shove user_id into a pattern that precludes future shared-document. Plan for `story_collaborators` table later (joins users to story-scoped resources). Don't build, but don't preclude.
8. **`scope-store` test impact.** Currently no test depends on the playhead store; verified via `grep -r '$playhead' tests/`. Adding a derived store is additive, not breaking. If new tests target scope, they go in `tests/unit/scope.test.ts`.

### Architecture
1. **`currentScope` derivation** — client-side only. Memoize by `(currentSceneId, intervalsVersion)` where `currentSceneId` is a cheap projection of `t` via binary search of scene boundaries (NOT an output of the scope computation; it's a bucket of the input). `intervalsVersion` increments on every `$intervals.update()`. During a playhead drag, intervals don't mutate so the memo holds across frames; it invalidates only on a real intervals edit. **Test:** also assert memo holds across a page reload after `intervalsVersion` resets to 0 (the cache must rebuild correctly from a fresh start).
