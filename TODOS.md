### 3. Svelte Stores + Cross-Window Navigation

- [x] `src/lib/stores/entities.ts` — writable store, load on app init, optimistic update helpers (`createEntity`, `updateEntity`, `deleteEntity`)
- [x] `src/lib/stores/relationships.ts` — same pattern; relationships carry semantic `type` values: `"appears_in"`, `"takes_place_at"`, `"caused_by"`, `"allied_with"`, `"rivals"`, `"mentor_of"`, `"located_at"`
- [x] `src/lib/navigation.ts` — `openEntity(id: string)` universal navigation function. One window per entity, keyed by `${appId}-${entityId}`. If the window already exists, bring it to front. If not, open a new instance. This means Elara's Character Editor and Damien's Character Editor are independent windows; Arc 1 Timeline and Arc 2 Timeline are independent windows; all can be open simultaneously.
  ```ts
  const ENTITY_APP: Record<EntityType, AppId> = {
    Character: 'character-editor',
    Location:  'world-map',
    Event:     'timeline',
    Act:       'timeline',
    Scene:     'timeline',
    Note:      'wiki',
  };

  export function openEntity(id: string) {
    const entity = get(entities).find(e => e.id === id);
    const appId = ENTITY_APP[entity.type];
    const windowId = `${appId}-${id}`;          // unique per entity
    const existing = get(windows).find(w => w.id === windowId);
    if (existing) {
      windowStore.focus(windowId);               // bring to front, no duplicate
    } else {
      windowStore.open({ id: windowId, appId, entityId: id }); // new window
    }
  }
  ```
- [x] `windowStore` tracks windows as a list of `{ id, appId, entityId, x, y, width, height, minimized, zIndex }`. Windows are keyed by `id` (`${appId}-${entityId}`), not by `appId` — multiple instances of the same app type are fully supported.
- [x] Taskbar: group open windows by `appId`. If only one Character Editor is open, show one button. If three are open, show a grouped button with a count badge — clicking it opens a mini-picker listing Elara, Damien, Scout. Prevents taskbar clutter with many open entity windows.

## Phase 2 — Desktop Shell

### 4. Window Manager

- [x] `src/lib/components/WindowManager.svelte` — renders all open windows, manages z-index stack
- [x] `src/lib/components/Window.svelte` — draggable, resizable, minimizable; props: `title`, `appId`, `initialPosition`
- [x] **Window spawn position:** Cascading offset. Track a module-level `spawnOffset` counter in `windowStore`. Each new window opens at `{ top: 60 + (spawnOffset % 8) * 24, left: 60 + (spawnOffset % 8) * 24 }`, then increment counter. Resets modulo 8 so it doesn't push off a 1280px viewport. Default window size: 320×480px.
- [x] `src/lib/stores/windows.ts` — open/close/focus/minimize state for all windows
- [x] `src/lib/components/Taskbar.svelte` — app buttons, minimized window list, story position slot (placeholder)
- [x] `src/lib/components/Desktop.svelte` — background, icon dock
- [x] **Minimum viewport:** Below 1280px wide, replace the entire app with a centered lockout message: "betwixt-and-between works best on a larger screen. Expand your browser window to continue." Implemented as a CSS media query wrapper + a `<TooSmall>` component — no JS required.
- [x] **Keyboard shortcuts** (handle in `WindowManager.svelte` via `keydown` on `document`):
  - `Ctrl+W` — close the currently focused window
  - `Ctrl+Tab` / `Ctrl+Shift+Tab` — cycle forward/backward through open windows (update `zIndex` stack)
  - `Escape` — dismiss any open inline form (relationship label prompt, etc.)
  - `Tab` / `Shift+Tab` — navigate between EntityLink chips within a section
  - `Enter` — activate the focused chip (calls `openEntity(id)`)
  - `Ctrl+K` — reserved for future command palette (no-op for now, don't let browser default handle it)
- [x] **Focus rings:** All interactive elements (chips, buttons, window controls) must have a visible focus ring using `outline: 2px solid var(--color-accent)` — never `outline: none` without a replacement.
- [x] **First-open / empty state:** Clean dark desktop with app icons in the dock. No windows open on fresh launch. A one-time tutorial overlay identifies each dock icon with a label + one-line description (e.g. "Characters — manage your cast", "Story Graph — see how they connect"). Overlay is dismissible and never shows again after first close. The empty desktop itself shows a subtle centered line: "Your story starts here." (disappears once the first entity is created). This preserves the OS aesthetic while orienting new users.
- [x] Playwright tests:
  - Open window → appears on screen
  - Click taskbar button → window comes to front (z-index check)
  - Drag window → position updates
  - Minimize → window hidden, taskbar button still present

## Phase 5 — Architecture Rework (post-v0.1.0)

Generated by /plan-eng-review on 2026-04-28. Source of truth: design doc at `~/.gstack/projects/jzeng151-betwixt-and-between/steve-feat-timeline-redesign-design-20260428-111518.md` and `CONSIDERATIONS.md` in repo root.

### 10. PR 1 — Intervals schema + migration + compat layer ✅ SHIPPED 2026-04-28

Schema is the load-bearing piece. Ship first, derisk everything else.

- [x] **Schema delta on `entities`:** ALTER TABLE add `parent_id TEXT REFERENCES entities(id) ON DELETE CASCADE` and `position INTEGER`, both nullable. Drizzle migration step. Acts at root level use `position` for sibling ordering at the type-level — see CONSIDERATIONS.md "/plan-eng-review resolutions" item 5.
- [x] **New `intervals` table** with hybrid FK + computed-position columns. Schema sketched in CONSIDERATIONS.md → "Premise 4." `CHECK (start_position < end_position)` strict. Composite index on `(start_position, end_position)`, plus `(entity_id)` for renderer lookups.
- [x] **Pre-migration snapshot via `sqlite3 .backup`** (one-line shell wrap; restore is `cp file.backup local.db`). Documented in CONSIDERATIONS.md → "Migration: rollback and recovery."
- [x] **`computeIntervalPositions(input)`** pure function in `src/lib/server/intervals.ts`. Source-of-truth for the position math. Unit tests in `test/intervals-math.test.ts` covering all six worked-example cases from CONSIDERATIONS.md.
- [x] **`writeInterval(input)`** chokepoint in `src/lib/server/intervals.ts`. Validates polymorphic FK types (Act / Scene), validates positions match `computeIntervalPositions()` if both supplied, computes positions if only FKs supplied, writes in a transaction. CI grep prevents raw inserts/updates outside this function.
- [x] **`recomputeIntervalsForAct(actId)`** branches on scene FK presence: scene-anchored intervals follow their scene; fraction-positioned (no scene FK) stay frozen. Runs in same transaction as Scene mutations.
- [x] **`recomputeAllIntervals()`** runs in same transaction as any Act-level mutation (insert/reorder/delete). Walks every interval row and recomputes positions from FKs. Without this, inserting an Act between two existing acts silently shifts every downstream position.
- [x] **Migration script** (`scripts/migrations/intervals-migration.ts`): walks every `appears_in` relationship, validates Character → Act shape, calls `writeInterval()` for each, deletes successfully-migrated rows from `relationships`. Malformed rows logged to `migration-warnings.txt`, skipped (do not abort). Idempotent across fresh-install (no `appears_in`) and existing-data paths.
- [x] **`src/lib/server/timeline-compat.ts`** with TWO functions: `getCharacterAppearancesForActs()` for V1 timeline data loader, AND `getAppearsInRelationships()` for StoryGraph (synthesizes `appears_in`-shaped rows from intervals on read). Symmetric with the write hijack. Throwaway code, deleted in PR 2.
- [x] **Hijack** `POST /api/relationships` with `type='appears_in'`: internally call `writeInterval()`. Existing UIs (CharacterEditor "Linked Entities," StoryGraph edge-creation) keep working transparently. ~20 lines, deleted in PR 2.
- [x] **API endpoints:** `src/routes/api/intervals/+server.ts` (GET list with optional `?entity_id=` filter, POST create). `src/routes/api/intervals/[id]/+server.ts` (GET single, PATCH partial update, DELETE).
- [x] **Vitest invariant test** at `test/db-invariants/intervals.test.ts`: walks every interval row, asserts (a) FK type alignment (start_act_id resolves to type=Act, etc.), (b) position-FK consistency for scene-anchored rows (positions match `computeIntervalPositions()` derivation within float epsilon), (c) Act-ordering consistency (`floor(start_position) == act_index_of(start_act_id)`).
- [x] **Critical regression tests:** add character to act via CharacterEditor → appears on V1 timeline. Add character to act via StoryGraph edge → appears on V1 timeline. Migration is idempotent. Malformed `appears_in` migration policy. Existing 41 E2E tests pass post-migration. Delete-Act cascade smoke test (intervals + scenes both gone, no orphans).
