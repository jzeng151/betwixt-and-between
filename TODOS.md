- [x] `src/lib/stores/relationships.ts` ‚Äî same pattern; relationships carry semantic `type` values: `"appears_in"`, `"takes_place_at"`, `"caused_by"`, `"allied_with"`, `"rivals"`, `"mentor_of"`, `"located_at"`
- [x] `src/lib/navigation.ts` ‚Äî `openEntity(id: string)` universal navigation function. One window per entity, keyed by `${appId}-${entityId}`. If the window already exists, bring it to front. If not, open a new instance. This means Elara's Character Editor and Damien's Character Editor are independent windows; Arc 1 Timeline and Arc 2 Timeline are independent windows; all can be open simultaneously.
  ```ts
  const ENTITY_APP: Record<EntityType, AppId> = {
    Character: 'character-editor',
    Location:  'world-map',
    Event:     'timeline',
    Act:       'timeline',
    Scene:     'timeline',
    Note:      'wiki',
  };

  export function openEntity(id: string) {
    const entity = get(entities).find(e => e.id === id);
    const appId = ENTITY_APP[entity.type];
    const windowId = `${appId}-${id}`;          // unique per entity
    const existing = get(windows).find(w => w.id === windowId);
    if (existing) {
      windowStore.focus(windowId);               // bring to front, no duplicate
    } else {
      windowStore.open({ id: windowId, appId, entityId: id }); // new window
    }
  }
  ```
- [x] `windowStore` tracks windows as a list of `{ id, appId, entityId, x, y, width, height, minimized, zIndex }`. Windows are keyed by `id` (`${appId}-${entityId}`), not by `appId` ‚Äî multiple instances of the same app type are fully supported.
- [x] Taskbar: group open windows by `appId`. If only one Character Editor is open, show one button. If three are open, show a grouped button with a count badge ‚Äî clicking it opens a mini-picker listing Elara, Damien, Scout. Prevents taskbar clutter with many open entity windows.

### 10. PR 1 ‚Äî Intervals schema + migration + compat layer ‚úÖ SHIPPED 2026-04-28

Schema is the load-bearing piece. Ship first, derisk everything else.

- [x] **Schema delta on `entities`:** ALTER TABLE add `parent_id TEXT REFERENCES entities(id) ON DELETE CASCADE` and `position INTEGER`, both nullable. Drizzle migration step. Acts at root level use `position` for sibling ordering at the type-level ‚Äî see CONSIDERATIONS.md "/plan-eng-review resolutions" item 5.
- [x] **New `intervals` table** with hybrid FK + computed-position columns. Schema sketched in CONSIDERATIONS.md ‚Üí "Premise 4." `CHECK (start_position < end_position)` strict. Composite index on `(start_position, end_position)`, plus `(entity_id)` for renderer lookups.
- [x] **Pre-migration snapshot via `sqlite3 .backup`** (one-line shell wrap; restore is `cp file.backup local.db`). Documented in CONSIDERATIONS.md ‚Üí "Migration: rollback and recovery."
- [x] **`computeIntervalPositions(input)`** pure function in `src/lib/server/intervals.ts`. Source-of-truth for the position math. Unit tests in `test/intervals-math.test.ts` covering all six worked-example cases from CONSIDERATIONS.md.
- [x] **`writeInterval(input)`** chokepoint in `src/lib/server/intervals.ts`. Validates polymorphic FK types (Act / Scene), validates positions match `computeIntervalPositions()` if both supplied, computes positions if only FKs supplied, writes in a transaction. CI grep prevents raw inserts/updates outside this function.
- [x] **`recomputeIntervalsForAct(actId)`** branches on scene FK presence: scene-anchored intervals follow their scene; fraction-positioned (no scene FK) stay frozen. Runs in same transaction as Scene mutations.
- [x] **`recomputeAllIntervals()`** runs in same transaction as any Act-level mutation (insert/reorder/delete). Walks every interval row and recomputes positions from FKs. Without this, inserting an Act between two existing acts silently shifts every downstream position.
- [x] **Migration script** (`scripts/migrations/intervals-migration.ts`): walks every `appears_in` relationship, validates Character ‚Üí Act shape, calls `writeInterval()` for each, deletes successfully-migrated rows from `relationships`. Malformed rows logged to `migration-warnings.txt`, skipped (do not abort). Idempotent across fresh-install (no `appears_in`) and existing-data paths.
- [x] **`src/lib/server/timeline-compat.ts`** with TWO functions: `getCharacterAppearancesForActs()` for V1 timeline data loader, AND `getAppearsInRelationships()` for StoryGraph (synthesizes `appears_in`-shaped rows from intervals on read). Symmetric with the write hijack. Throwaway code, deleted in PR 2.
- [x] **Hijack** `POST /api/relationships` with `type='appears_in'`: internally call `writeInterval()`. Existing UIs (CharacterEditor "Linked Entities," StoryGraph edge-creation) keep working transparently. ~20 lines, deleted in PR 2.

## Future feature: temporal identity & relationships

Surfaced 2026-05-01 while seeding *The Prestige* ‚Äî see
`scripts/seed/prestige.ts`. The current data model is **static**: a
relationship row exists or it doesn't. Stories have identities
and allegiances that **change across story-time**.

Examples in *The Prestige* alone:
- Olivia Svenson is `allied_with` Angier in Act 2 ‚Üí switches sides ‚Üí `allied_with` Borden by Act 3. Both are true at different points; today the seed picks one arbitrarily and the graph misrepresents the trajectory.
- Rupert Angier is privately `Lord Caldlow` from birth, but the noble identity is **secret** until the Caldlow vault scene in Act 5. A reader who hasn't reached Act 5 should not see the alias.
- Borden's twin pact: Frederick Borden is `allied_with` Alfred Borden privately throughout, but the alliance is invisible to all other characters. The graph today has no concept of "visible to whom."

Pending design (no PR):

- **Temporal validity on relationships:** add optional `start_act_id` / `end_act_id` (or `start_position` / `end_position` once intervals' positioning system is generalized) to relationship rows. NULL = "always true."
- **Spotlight integration:** when the timeline scrubber sits at story-time `t`, the StoryGraph hides relationships whose `[start_act, end_act]` interval doesn't contain `t`. This subsumes the current static-graph view (every rel is "always true") as the special case.
- **Reveal gating:** distinct from temporal validity ‚Äî a relationship can be true the whole novel but only revealed at a specific story moment. Probably a separate `revealed_at_event_id` field. Out of scope for v1.
- **Identity aliases:** tracked as a new entity-pair table (`entity_aliases (primary_id, alias_id, revealed_at_event_id)`)? Or as a relationship type (`aka`)? Open question.

Big feature. Re-visit when the seeded *Prestige* dataset surfaces concrete UX pain that the static graph can't represent.
